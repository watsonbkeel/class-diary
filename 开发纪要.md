# 班级日记本小程序开发纪要

> 本纪要总结“班级日记本”小程序从生成、启动、联调到体验优化的主要开发过程，重点记录复杂度高、容易出错、反复修改的功能点，方便后续维护。

## 1. 项目阶段概览

### 1.1 完整项目生成阶段

初始目标是一次性生成完整项目，包括：

- 微信小程序原生前端。
- Node.js Express 后端。
- Prisma + MySQL 数据库。
- Docker Compose MySQL。
- JWT 账号密码认证。
- 本地图片上传。
- 通知、举报、点赞、评论、匿名、管理员管理。

这一阶段的重点不是 UI，而是：

- 项目能跑。
- 接口闭环。
- 权限正确。
- 匿名展示正确。
- 管理员可以追溯匿名作者。

初始生成后经过多轮检查，修复了 Prisma 关系、通知字段、匿名真实作者返回位置、WXML 表达式等问题。

### 1.2 本地启动与数据库阶段

启动时遇到两个关键问题：

- Prisma `latest` 安装到了 v7，和当前 schema 的 datasource `url` 写法不兼容。
- MySQL 用户缺少 Prisma migrate 创建 shadow database 的权限。

最终处理：

- 将 Prisma 固定到 v6 稳定版本。
- 给本地开发数据库用户补足迁移权限。
- 完成 MySQL Docker、Prisma migrate、后端启动和 `/health` 验证。

### 1.3 微信教育小程序 + frp 联调阶段

部署环境是 Debian 服务器，微信开发者工具在远程 Mac。

核心问题：

- 小程序原配置使用 `127.0.0.1`，Mac 和手机扫码都访问不到 Debian 的本机地址。
- 用户使用 frp 暴露 `3000` 端口为 `http://bkeel.com:5300`。
- 教育体验版获取真实 `WECHAT_SECRET` 成本较高，后续产品方向改为账号名 + 密码注册登录。

最终处理：

- 小程序 `PUBLIC_BASE_URL` 和 `API_BASE_URL` 改为 `http://bkeel.com:5300`。
- 后端 `.env` 的 `PUBLIC_BASE_URL` 也同步。
- 验证公网 `/health`、账号注册/登录、带 token 的 `/api/classes/my` 均可用。

### 1.4 业务体验修复阶段

上线体验后，根据真实使用反馈持续修复：

- 创建班级权限下放。
- 输入框窄、看不清文字。
- 用户昵称和班级昵称。
- 图片他人不可见。
- 新手指引位置。
- Bee 上传图片转圈。
- 图片点击全屏预览。
- 上传目录按周分文件夹。
- 通知点击进入对应日记。

## 2. 复杂度较高、容易出错的区域

### 2.1 权限模型

权限是整个项目最容易出错的部分，涉及：

- 登录态。
- 是否为班级成员。
- 是否审核通过。
- 是否禁言。
- 是否为班级管理员。
- 是否为内容作者。
- 匿名内容是否允许返回真实作者。

容易出错点：

- 普通接口不能返回匿名真实作者。
- 管理接口可以返回匿名真实作者。
- 管理员只能管理自己是 admin 的班级。
- 禁言用户不能发日记、评论、回复。
- 删除/隐藏不能物理删除。
- 不能移除/禁言自己。
- 不能取消/移除最后一个管理员。

本项目中实际修复过的问题：

- 匿名真实作者一开始嵌套在 `author` 里，后来改成管理员才看到的顶层 `realAuthor`。
- 匿名评论通知需要避免泄露真实触发者。
- 评论删除/隐藏时需要同步处理回复和 `Diary.commentCount`。
- 班级创建最初依赖全局管理员，后来根据需求改为任何登录用户都可创建班级，创建者自动成为该班 admin。

维护建议：

- 后续新增任何接口都先确认是否需要 `ensureClassMember`、`ensureClassAdmin`、`ensureNotMuted`。
- 匿名相关字段统一走 service 层 sanitize，不要在 controller 或前端临时判断。
- 管理员视角和普通成员视角最好用接口测试覆盖。

### 2.2 Prisma + MySQL 数据模型

复杂点：

- Prisma 和 MySQL enum、关系、唯一索引需要匹配业务。
- Like、Report、ModerationLog 使用多态 `targetType + targetId`，不能直接建传统 Prisma relation。
- Prisma v7 与旧 datasource 写法不兼容。
- Prisma migrate 在 MySQL 上需要 shadow database 权限。

实际问题：

- 初始代码尝试把多态 Like 当成普通 relation include，导致 Prisma 关系不成立。
- JoinRequest 如果用过强唯一约束，会阻止历史申请后再次申请。
- Notification 字段最初与文档不一致，后来统一为 `userId/actorId`。
- 后续新增 `ClassMember.classNickname` 需要迁移和 Prisma Client regenerate。

维护建议：

- 不要把多态目标强行建 relation。
- 新增字段时必须同时：改 schema、写 migration、执行 migrate、执行 `npx prisma generate`。
- 修改 schema 后要跑 `npx prisma validate`。

### 2.3 微信小程序 WXML/JS 限制

复杂点：

- WXML 不支持任意 JS 函数调用。
- 页面里不能像 Vue/React 一样随意计算表达式。
- 复杂展示字段需要在 JS 中预处理。
- `wx.navigateTo`、`wx.reLaunch`、返回栈行为必须明确。

实际问题：

- 初始 WXML 使用类似 `{{formatDate(...)}}` 的函数调用，后来改成 JS 里预计算 `createdAtText`。
- 图片全屏预览需要提前准备 `imagePreviewUrls`。
- 通知跳转日记详情需要用 `wx.navigateTo`，不能用 `redirectTo` 或 `reLaunch`，否则返回不到通知页。
- 通知关联评论需要传 `commentId`，详情页再高亮对应评论/回复。

维护建议：

- 页面模板只放简单字段展示，复杂逻辑放 `prepareXxx()`。
- 涉及返回路径时优先用 `navigateTo` 保留栈。
- 修改 WXML 后尽量在微信开发者工具里实际编译确认。

### 2.4 图片上传与展示

这是整个项目中反复修改最多的区域之一。

涉及链路：

1. 小程序 `wx.chooseMedia` 选择图片。
2. 小程序 `wx.uploadFile` 上传到后端。
3. 后端 Multer 保存到 `server/uploads`。
4. 上传接口返回相对路径和完整 URL。
5. 发日记时把图片 URL 写入 `DiaryImage`。
6. 列表/详情接口返回图片。
7. 小程序渲染图片。
8. 点击图片全屏预览。

实际问题和修复：

- 上传返回字段最初存在 `path/url` 不一致，后来统一使用 `url`。
- 早期返回绝对 URL，后来改回相对 `/uploads/...`，再额外返回 `fullUrl`。
- 他人看不到图片时，确认服务端和数据库都正常，最终增加前端 image cache，用 `wx.request` 拉取二进制写入本地文件显示。
- Bee 上传图片一直转圈，最终增加压缩、超时、重试、进度文案、错误提示、HEIC 拦截、5MB 检查、上传期间锁定控件。
- 图片点击不能放大，最后接入 `wx.previewImage`。
- 上传目录从扁平 `uploads/` 改为按 ISO 周分目录：`uploads/YYYY-Www/`。

复杂原因：

- 微信开发者工具、真机、教育体验版对 HTTP、域名、图片展示的行为可能不同。
- iPhone HEIC/HEIF、图片体积、Multer fileFilter、网络超时都可能导致“转圈后恢复原状”。
- 上传成功和发日记成功是两个不同阶段，需要分开提示。

维护建议：

- 保持上传接口返回：`{ url, fullUrl }`。
- 数据库存相对路径，前端展示优先使用后端 `fullUrl`。
- 移动端上传失败必须显示明确错误，不要静默 catch。
- 上传 UI 必须防重复点击。
- 如果后续正式发布，最好换 HTTPS 域名，减少微信真机图片域名问题。

### 2.5 通知系统

复杂点：

- 通知由点赞、评论、回复、审核、举报处理多处创建。
- 不同通知的可跳转目标不同。
- 匿名评论/回复通知不能泄露真实 actor。
- 用户希望从通知直接进入上下文继续互动。

实际问题和修复：

- 通知数据已经有 `diaryId/commentId/actor/class`，但前端一开始没有点击行为。
- 后来通知卡片支持点击，有 `diaryId` 时进入日记详情。
- 未读通知点击时先标记已读。
- `commentId` 传入详情页后高亮评论/回复。
- 详情页顶部显示来源提示，如“Bee 评论了这篇日记”。

维护建议：

- 新增通知类型时确认是否有关联 `diaryId/commentId`。
- 对不可跳转通知显示“暂无可跳转内容”。
- 不要用 `reLaunch` 打开详情，否则无法返回通知页。

## 3. 反复修改次数较多的功能

### 3.1 班级创建

修改过程：

1. 初始规则：第一个全局管理员创建第一个班级。
2. 用户反馈新用户无班级时没有创建入口。
3. 增加无班级时创建入口。
4. 用户进一步要求取消全局管理员限制。
5. 最终规则：任何登录用户都可创建班级，创建者自动成为该班级管理员。

高风险点：

- 文档原规则和后续真实需求发生变化。
- UI 是否显示创建入口与后端是否允许创建必须一致。

### 3.2 图片功能

修改过程：

1. 基础上传。
2. 返回相对路径。
3. 前端拼接 `PUBLIC_BASE_URL`。
4. 他人不可见，增加 `fullUrl`。
5. 仍不可见，增加本地 image cache。
6. Bee 上传转圈，增加上传可靠性和 UX。
7. 图片点击放大，接入 `wx.previewImage`。
8. 上传目录按周分文件夹。

高风险点：

- 任何一处路径规则变化都会影响上传、存库、展示、预览。
- 真机和开发者工具表现可能不同。
- HTTP/frp 在教育体验版可用，但长期仍建议 HTTPS。

### 3.3 昵称系统

修改过程：

1. 登录默认显示“微信用户”。
2. 用户要求可修改昵称。
3. 增加全局昵称编辑。
4. 用户希望不同班级不同昵称。
5. 增加 `ClassMember.classNickname`。
6. 日记/评论展示优先使用班级昵称。

高风险点：

- 登录时不能因为空昵称把已有昵称覆盖回“微信用户”。
- 匿名内容仍必须显示“匿名同学”，不能因为班级昵称泄露作者。

### 3.4 新手指引

修改过程：

1. 登录页和班级选择页加入新手指引。
2. 用户反馈位置遮挡老用户视线。
3. 将新手指引移动到已加入班级列表之后、创建班级之前。

高风险点：

- 新用户需要一眼看到。
- 老用户不能被高频提示打断主操作。

### 3.5 通知跳转

修改过程：

1. 通知只支持列表和已读。
2. 用户要求点击通知进入日记。
3. 增加通知卡片点击。
4. 增加日记详情来源提示。
5. 增加评论高亮。

高风险点：

- 必须保留返回栈。
- 点赞通知没有 `commentId`，评论通知有 `commentId`，两者处理不同。

## 4. 重要设计决策

### 4.1 从微信 mock 登录切换为账号密码登录

原因：

- 用户没有找到教育小程序 AppSecret。
- 微信 mock 登录会导致不同设备或不同 code 生成不同身份，老用户可能找不到原来的班级。
- 教育体验阶段更需要稳定账号体系，优先验证业务闭环。

影响：

- 当前版本首次打开必须注册账号名和密码。
- 老用户换设备后只要使用同一个账号名和密码登录，就能找回原班级。
- `/api/auth/wechat-login` 保留兼容路由，但返回 `WECHAT_LOGIN_DISABLED`，提示使用账号密码登录。

### 4.2 教育体验版先用 frp HTTP

原因：

- 用户使用微信教育小程序简单上线/扫码体验。
- 教育场景对正式合法域名要求较低。
- 当前公网入口为 `http://bkeel.com:5300`。

影响：

- 可以快速体验。
- 但真机图片/请求仍可能受微信环境限制，长期建议使用 HTTPS。

### 4.3 数据库存相对图片路径

原因：

- 避免公网域名变化时需要迁移数据库。
- 后端可根据当前 `PUBLIC_BASE_URL` 生成 `fullUrl`。

影响：

- `DiaryImage.url` 保存 `/uploads/...`。
- API 返回时附加 `fullUrl`。
- 前端优先使用 `fullUrl`，失败时可走本地缓存兜底。

### 4.4 服务端分层保持稳定

项目坚持：

- routes 只绑定路由。
- controllers 解析请求和返回响应。
- services 承载业务逻辑。
- middleware 处理鉴权、错误、上传。
- utils 放通用工具。

效果：

- 后续多次功能调整主要集中在 service/page 层，没有打乱整体结构。

## 5. 验证方式总结

开发中反复使用的验证方式：

- `node --check` 检查 JS 语法。
- LSP diagnostics 检查明显错误。
- `npx prisma validate` 检查 Prisma schema。
- `npx prisma migrate deploy` / `migrate status` 检查数据库迁移。
- `curl http://bkeel.com:5300/health` 检查公网后端。
- 账号密码注册/登录、班级、上传、图片、通知链路。
- 用真实数据库用户 Bee、Cookie、Watson 等现有账号做业务链路验证。
- 使用 Oracle 做关键实现的只读审查。

## 6. 后续维护建议

### 6.1 优先补充自动化测试

目前项目已通过手工/API 验证，但后续建议补充：

- 权限测试。
- 匿名字段测试。
- 通知创建测试。
- 上传路径测试。
- 班级昵称展示测试。

### 6.2 图片服务建议升级到 HTTPS

当前 `http://bkeel.com:5300` 在教育体验版可用，但图片显示已经多次受环境影响。

建议后续：

- 使用正式域名。
- 配置 HTTPS。
- 反代到本地 `3000`。
- 将 `PUBLIC_BASE_URL`、`API_BASE_URL` 改为 HTTPS。

### 6.3 管理功能后续要加强审计

目前已有 ModerationLog，但后续可增强：

- 管理员操作列表页。
- 举报处理历史。
- 被禁言原因展示。
- 成员移除记录。

### 6.4 上传清理和容量管理

目前上传按周分目录，但没有清理策略。

建议后续：

- 定期统计 `uploads/` 大小。
- 对软删除内容关联图片做清理策略。
- 保留备份后再删除。

### 6.5 文档需要随接口变化同步

这次新增过：

- `PATCH /api/auth/me`
- `POST /api/auth/register` / `POST /api/auth/login`
- `GET /api/classes` 公开班级列表
- `PATCH /api/classes/:classId/nickname`
- 图片按周目录
- 通知点击跳转

后续每次新增接口都要同步 `docs/API_SPEC.md` 和 README 相关说明。

## 7. 当前项目状态摘要

截至本纪要生成时：

- 后端运行在 `http://127.0.0.1:3000`，通过 frp 暴露为 `http://bkeel.com:5300`。
- MySQL 运行在 Docker。
- 小程序 API 指向 `http://bkeel.com:5300/api`。
- 当前版本使用账号名 + 密码注册登录，旧微信登录接口已禁用。
- 班级、日记、评论、点赞、通知、举报、管理、上传链路均已实现。
- 图片已支持显示、缓存兜底、全屏预览和按周目录存储。
- 通知已支持点击进入日记详情并返回通知页。
